# AllCanCode 项目目标

## 1. 项目总目标

在当前仓库基础上，建设一套可重新部署的 `allcancode + sub2api` 双应用体系：

- 普通用户只访问 `allcancode`
- 管理员只访问 `sub2api` 管理端
- 网关转发能力继续保留在 `sub2api`
- 整体以 Docker Compose 方式完成本地和后续服务器部署

本项目当前不再追求对“已运行中的旧版 `sub2api` 应用”的无缝增强升级，而是以“整套重新部署、结构清晰、后续易演进”为核心原则推进。

## 2. 产品边界

### 2.1 `allcancode`

`allcancode` 是新的普通用户端应用，负责承接所有用户侧核心功能。

必须承载：

- 登录
- 注册
- 退出登录
- 当日使用总览
- Key 管理
- 使用记录明细
- 消费记录
- 充值与支付
- 订单记录
- 用户基础账户中心

普通用户不再直接进入 `sub2api` 后端用户页面或管理页面。

### 2.2 `sub2api`

`sub2api` 继续承担后台管理与网关核心职责。

保留职责：

- 管理员登录
- 管理员后台
- 渠道、模型、供应商、路由配置
- 网关转发
- 模型访问控制
- 平台级运营配置

管理员继续通过 `sub2api` 管理端完成后台管理工作，不迁移到 `allcancode`。

## 3. 本次交付范围

首个稳定版本至少实现以下普通用户功能：

- 登录
- 注册
- 获取当前用户信息
- 当日使用总览
- API Key 列表
- API Key 创建
- API Key 备注修改
- API Key 启用与禁用
- API Key 删除
- 使用记录明细查询
- 消费记录查询
- 充值订单创建
- 支付方式选择
- 支付状态轮询或回调完成
- 订单历史

本次明确不做：

- 对旧版正在运行的 `sub2api` 做原地增强
- 将管理员后台整体迁入 `allcancode`
- 复杂营销、分销、推广功能
- 过度微服务化拆分

## 4. 总体架构目标

### 4.1 目标服务拓扑

最终部署拓扑至少包含：

- `allcancode-web`
- `allcancode-api`
- `sub2api`
- `postgres`
- `redis`
- `nginx`

其中：

- `allcancode-web` 负责普通用户前端页面
- `allcancode-api` 负责普通用户侧 BFF 和业务接口
- `sub2api` 负责后台管理与网关能力
- `postgres`、`redis` 满足 `sub2api` 及新系统运行依赖
- `nginx` 负责统一入口和路由分发

### 4.2 统一路由约定

推荐并固化以下访问分流：

- `/` -> `allcancode-web`
- `/api/app/*` -> `allcancode-api`
- `/admin/*` -> `sub2api`
- `/v1/*` -> `sub2api`
- 支付回调相关路由 -> `allcancode-api`

这个路由约定视为系统长期契约，后续升级尽量不破坏。

## 5. 前后端边界原则

### 5.1 前端只依赖 `allcancode-api`

`allcancode` 前端必须只调用 `/api/app/*`。

不允许：

- 前端直接混用 `sub2api` 的普通用户接口
- 前端直接依赖 `sub2api` 管理端接口
- 在页面层拼接多个底层后台服务契约

### 5.2 `allcancode-api` 作为普通用户统一后端

`allcancode-api` 是普通用户端唯一后端入口，负责：

- 用户认证
- 用户信息聚合
- Dashboard 聚合接口
- Key 管理编排
- 使用记录查询
- 消费记录查询
- 支付配置下发
- 订单创建与状态查询
- 支付回调处理
- 对 `sub2api` 的封装集成

### 5.3 `sub2api` 继续做核心能力服务

`sub2api` 继续负责：

- 管理员后台
- 网关核心转发
- 模型渠道和供应商管理
- 系统级控制逻辑

## 6. 数据与集成策略

### 6.1 集成优先级

普通用户能力接入优先遵循以下顺序：

1. 通过 `allcancode-api` 封装 `sub2api` 已有普通用户能力
2. 对用户产品增强所需的新数据，由 `allcancode-api` 自己维护扩展表
3. 尽量避免前期就深度耦合 `sub2api` 内部表结构

### 6.2 数据归属原则

`sub2api` 继续作为以下数据的主权威来源：

- 核心用户账户数据
- API Key
- 网关使用日志
- 模型访问控制
- 渠道、供应商、路由、配额等网关核心数据

`allcancode-api` 适合承载以下扩展数据：

- 用户侧 Dashboard 聚合结果或缓存投影
- 用户展示用支付方式目录
- 支付方式与上游支付通道绑定关系
- 支付回调审计记录
- 对账与补单记录
- 退款申请记录
- 用户侧支付展示元数据

### 6.3 设计原则

如果数据是为了“网关运行”，应尽量留在 `sub2api`。

如果数据是为了“普通用户产品体验、支付编排、用户账单呈现、回调审计”，应优先由 `allcancode-api` 承担。

## 7. 普通用户端功能设计目标

### 7.1 Dashboard

当日使用总览页至少提供：

- 今日消耗金额或额度
- 今日请求次数
- 当前余额或账户额度
- Key 数量摘要
- 最近消费趋势
- 最近充值或订单摘要

Dashboard 应由后端聚合后一次返回，不让前端自行拼装多个底层接口。

### 7.2 Key 管理

普通用户必须能够：

- 查看 Key 列表
- 创建 Key
- 修改 Key 备注
- 启用或禁用 Key
- 删除 Key
- 查看脱敏后的 Key 信息和元数据

后台级高级控制仍保留在 `sub2api` 管理端。

### 7.3 使用记录

使用记录至少支持：

- 时间筛选
- 模型筛选
- Key 筛选
- 分页
- 汇总统计

### 7.4 消费记录

消费记录页至少能清晰区分：

- 模型调用扣费
- 充值入账
- 人工调账
- 退款流水
- 待支付或失败订单

### 7.5 订单与充值

充值流程至少支持：

- 创建订单
- 选择支付方式
- 发起跳转支付或二维码支付
- 支付回调或轮询更新状态
- 查询历史订单

## 8. 支付系统目标

支付不是单页面临时功能，而是明确的业务子系统。

### 8.1 支付接入目标

本项目支付侧要支持：

- 余额充值
- 套餐购买
- 易支付类接入方式
- 启润支付兼容接入
- 后续新增其他支付通道时不需要重写前端

### 8.2 启润支付策略

参考启润支付文档与 `sub2api` 现有支付配置，第一阶段采用“兼容易支付能力并做增强”的方式接入。

目标要求：

- 支持启润支付支持的全部可配置支付方式
- 支付方式列表由后端配置驱动，不在前端写死
- 签名、商户号、密钥、回调校验仅保留在后端
- 支付通道差异收敛到支付适配器层
- UI 展示信息和上游支付方式编码解耦

### 8.3 支付抽象层设计

支付模块至少要抽象三层：

1. 支付提供方实例
2. 支付方式目录
3. 支付方式绑定关系

建议结构：

#### 支付提供方实例

字段可包含：

- provider_code
- display_name
- endpoint
- merchant_id
- secret_ref
- callback_config
- enabled
- extra_config_json

#### 支付方式目录

字段可包含：

- method_code
- method_name
- icon
- enabled
- display_order
- client_flags

#### 支付方式绑定

字段可包含：

- method_code
- provider_instance_id
- provider_method_code
- priority
- enabled
- rule_json

### 8.4 退款能力目标

首版退款能力目标：

- 支持退款申请记录
- 支持退款状态跟踪
- 支持后台人工处理挂钩点
- 保留审计轨迹

首版不强制要求自动化退款执行。

## 9. 代码组织目标

### 9.1 前端

推荐保持：

- React
- TypeScript
- React Router
- 统一 API Client
- 统一认证状态管理
- 按业务模块划分页面与组件

### 9.2 后端

`allcancode-api` 应逐步从单文件演进为按模块组织：

- auth
- user
- dashboard
- keys
- usage
- billing
- payment
- orders
- system 或 config

每个模块后续自然演进为：

- handler/controller
- service
- repository 或 gateway adapter
- request/response model

### 9.3 数据库

数据库设计应遵循：

- 明确表归属
- 减少隐式耦合
- 有清晰初始化或 seed 策略
- 后续支持 migration 持续演进

## 10. 部署目标

### 10.1 Docker 部署要求

最终必须支持从零开始通过 Docker Compose 启动。

要求包括：

- 一条清晰的启动命令路径
- `.env` 驱动配置
- 明确的服务间 URL
- 仅后端注入敏感配置
- 合理 healthcheck
- 持久化卷
- 启动后尽量少人工操作

### 10.2 本地成功标准

本地 Docker 部署成功至少满足：

- 根路径可打开普通用户端
- `/api/app/*` 可正常访问
- `/admin/*` 可访问管理员端
- `/v1/*` 可访问网关相关接口
- `sub2api` 能依赖真实 `postgres`、`redis` 启动
- 支付回调路由可正确命中 `allcancode-api`

## 11. 实施阶段

### Phase 1：架构与部署骨架

- 明确服务边界
- 固化路由分流
- 接入真实 `sub2api + postgres + redis`
- 搭建 `allcancode-web` 与 `allcancode-api` 基础结构
- 搭建 Docker Compose 部署骨架

### Phase 2：普通用户核心能力

- 打通普通用户登录注册
- 打通用户信息接口
- 实现 Dashboard
- 实现 Key 管理
- 保证前端只依赖 `/api/app/*`

### Phase 3：记录与支付

- 实现使用记录
- 实现消费记录
- 实现订单历史
- 实现充值下单
- 实现启润支付兼容接入

### Phase 4：强化与沉淀

- 完成 `allcancode-api` 对 `sub2api` 的稳定封装
- 梳理支付配置和扩展模型
- 完善文档与 `.env.example`
- 完善部署说明与巡检方式
- 为后续迭代保留清晰扩展点

## 12. 验收标准

只有在以下条件都满足时，才算该目标完成：

- 普通用户只使用 `allcancode` 即可完成核心操作
- 管理员继续只使用 `sub2api` 管理端
- `allcancode` 已提供 Dashboard、Key 管理、使用记录、消费记录、充值、订单功能
- 支付方式由后端配置驱动，而不是前端写死
- 普通用户前端只依赖 `/api/app/*`
- 整个系统可以通过 Docker Compose 从零启动
- 拓扑中包含真实 `sub2api`、`postgres`、`redis`
- `/`、`/api/app/*`、`/admin/*`、`/v1/*` 路由职责清晰稳定
- 整体架构足够清晰，便于后续继续增强

## 13. 非目标

当前阶段明确不是目标的内容：

- 兼容旧版运行中 `sub2api` 的无缝升级
- 将管理员后台重写进 `allcancode`
- 搭建复杂微服务集群
- 一次性做完全部营销、推广、分销能力

## 14. 最终目标表述

构建一套“普通用户走 `allcancode`、管理员走 `sub2api`、网关继续由 `sub2api` 承担、支付可扩展、Docker 可重建部署、后续易升级”的清晰系统架构，并以此作为后续持续开发和交付的统一目标。
